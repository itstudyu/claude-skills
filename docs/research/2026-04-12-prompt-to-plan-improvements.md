## Executive Summary

4개 축(① Claude 4/4.6 prompt best practices, ③ Claude Code skills best practice,
② Extended Thinking & Plan Mode, ⑤ Subagent handoff)을 조사한 결과,

Claude 4.6 최적화와 공식 생태계 정합성이 크게 향상된다.

## Axis ①: Claude 4/4.6 Prompt Best Practices

Sources:
- [Claude 4 best practices](https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/claude-4-best-practices)
- [Prompt engineering overview](https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/overview)
- [Context engineering blog](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)

### Finding 1: Few-shot / Multishot examples missing

- **Current**: No before/after transformation examples in SKILL.md. Only one
  Flipped Interaction example (L98-108).
- **Official**: "Include 3-5 examples for best results. Wrap examples in `<example>`
  tags." — Anthropic prompt engineering guide.
- **Gap**: Without concrete input→output examples per tier, Claude has no reference
  for the expected transformation quality and format.
- **Proposal**: Add `<examples>` section with 3 before/after pairs (Lite/Standard/Deep).

### Finding 2: Claude 4.6 model-specific guidance missing

- **Current**: "Target Model Adaptation" (L371-376) only covers delimiter style
  (XML vs Markdown). No Claude 4.6-specific notes.
- **Official**: "Starting with Claude 4.6 models, prefilled responses on the last
  assistant turn are no longer supported." / "Prefer general instructions over
  prescriptive steps." / "Opus 4.6 has a tendency to overengineer."
- **Gap**: No guidance on: prefill being deprecated, overengineering tendency,
  adaptive thinking replacing manual CoT triggers.
- **Proposal**: Expand "Target Model Adaptation" with Claude 4.6 subsection covering
  these behavioral changes.

### Finding 3: WHY principle only partially applied

- **Current**: `<constraints>` section (L145-149) says "with WHY each matters" — good.
  But `<task>` and `<role>` template guidance lack WHY instruction.
- **Official**: "Providing context or motivation behind your instructions helps
  Claude better understand your goals." — Anthropic prompt engineering.
- **Gap**: WHY principle should extend to all template sections, not just constraints.
- **Proposal**: Add "include motivation/background" guidance to `<task>` template.

### Finding 4: "Tell what to do, not what not to do" principle missing

- **Current**: Anti-Patterns section (L346-370) has 17 "Do NOT" items.
  Template guidance doesn't mention positive phrasing preference.
- **Official**: "Tell Claude what to do instead of what not to do." — Anthropic.
- **Gap**: Refined specs generated by this skill may inherit the negative phrasing
  habit. No guidance to prefer positive instructions in generated specs.
- **Proposal**: Add template principle: "Prefer positive instructions over
  prohibitions in `<constraints>` and `<task>`."

### Finding 5: `<example>` / `<examples>` tag standard not taught

- **Current**: Template uses `<role>`, `<context>`, `<task>`, `<constraints>`,
  `<deliverables>`, `<acceptance_criteria>`. No `<examples>` section.
- **Official**: `<example>` / `<examples>` are standard Anthropic tags for few-shot.
- **Gap**: For complex transformation tasks in Deep tier, refined spec should
  optionally include `<examples>` section. Template doesn't mention this.
- **Proposal**: Add optional `<examples>` to Deep tier template.

### Finding 6: Long context "data top, query bottom" not explicit

- **Current**: Template order happens to put `<context>` above `<task>`, and L169
  has "Final query restating the core implementation ask" at bottom. Good, but
  accidental — not explained.
- **Official**: "Put longform data at the top. Queries at the end can improve
  response quality by up to 30%."
- **Gap**: No explicit design rationale for tag ordering.
- **Proposal**: Add brief note explaining why `<context>` is first and query is last.

### Finding 7: Spec self-check missing in REFINE phase

- **Current**: Plan self-review exists (L327-329). No self-check for the refined
  spec itself before presenting to user.
- **Official**: "Ask Claude to self-check. Append 'Before you finish, verify your
  answer against [test criteria].'"
- **Gap**: Spec could miss user intent without a verification step.
- **Proposal**: Add self-check between drafting and presenting the spec:
  "Verify all user intents are covered before presenting."

### Finding 8: Claude 4.6 overengineering/overthinking warning missing

- **Current**: L359 "Over-specify simple tasks" and L356 "Add constraints the user
  never implied" partially cover this.
- **Official**: "Opus 4.6 has a tendency to overengineer by creating extra files,
  adding unnecessary abstractions." / "Does significantly more upfront exploration
  than previous models."
- **Gap**: No explicit warning that refined specs targeting Claude 4.6 should include
  scope-limiting language to counteract overengineering tendency.
- **Proposal**: Add to Anti-Patterns or a new "Claude 4.6 Notes" section.

## Axis ③: Claude Code Skills Best Practice

Sources:
- [Claude Code Skills docs](https://code.claude.com/docs/en/skills)
- [Agent Skills best practices](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices)

### Finding 9: Description exceeds 250-char effective limit (~770 chars)

- **Current**: Description (L3-14) lists trigger phrases in 3 languages, ~770 chars.
- **Official**: "Descriptions longer than 250 characters are truncated in the skill
  listing." — Claude Code skills docs.
- **Gap**: Everything after 250 chars is invisible in skill discovery. Wasted tokens.
- **Proposal**: Compress description to <250 chars. Move trigger phrases to a
  separate `## Activation Triggers` section in SKILL.md body.

### Finding 10: Description uses imperative form, not third-person

- **Current**: L3 "Transform rough, informal user input..."
- **Official**: "Always write in third person." — Skills best practices.
  Example: "Processes Excel files and generates reports."
- **Gap**: Should be "Transforms..." (third-person -s form).
- **Proposal**: Change to third-person present tense.

### Finding 11: evals.json schema mismatch

- **Current**: Uses `"prompt"`, `"expected_output"` + `"expectations"`, no `"skills"`.
- **Official**: Schema uses `"query"`, `"expected_behavior"` (single array), `"skills"`.
- **Gap**: 4 field name mismatches with official eval schema.
- **Proposal**: Refactor evals.json to match official schema.

### Finding 12: HARD-GATE contradicts own anti-patterns

- **Current**: `<HARD-GATE>` (L35-44) uses "NEVER execute" — aggressive language.
  Anti-Patterns (L366) says don't use "IMPORTANT:", "CRITICAL:", "YOU MUST".
- **Official**: "Dial back aggressive language. Use more normal prompting." —
  Anthropic Claude 4 docs. No `HARD-GATE` pattern in official docs.
- **Gap**: Internal contradiction. HARD-GATE violates the skill's own rules.
- **Proposal**: Replace `<HARD-GATE>` with calm "Mandatory workflow gates" section.

### Finding 13: Progressive disclosure underused — 400-line monolith

- **Current**: All templates, protocols, edge cases in one 400-line file.
- **Official**: "Keep SKILL.md under 500 lines. Move detailed reference material
  to separate files." — Claude Code skills docs.
- **Gap**: At 400 lines, close to limit. Deep-tier-only content (research protocol,
  full template) loads every time even for Lite tier tasks.
- **Proposal**: Split into separate files: `templates.md`, `research-protocol.md`,
  `plan-conventions.md`. Reference from SKILL.md.

### Finding 14: allowed-tools grants Write/Edit/Bash without user confirmation

- **Current**: L15-26 lists Bash, Write, Edit among allowed-tools.
- **Official**: "allowed-tools grants permission so Claude can use them without
  prompting you for approval." — Claude Code skills docs.
- **Gap**: REFINE phase shouldn't need Write/Edit/Bash. These are only needed
  in PLAN execution, which should require user approval for safety.
- **Proposal**: Remove Write, Edit, Bash from allowed-tools. Keep Read, Grep, Glob,
  WebSearch, WebFetch, EnterPlanMode, ExitPlanMode, AskUserQuestion.

## Axis ②: Extended Thinking & Plan Mode

Sources:
- [Extended Thinking docs](https://platform.claude.com/docs/en/build-with-claude/extended-thinking)
- [Adaptive Thinking docs](https://platform.claude.com/docs/en/build-with-claude/adaptive-thinking)
- [Extended thinking tips](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/extended-thinking-tips)
- [Claude Code common workflows](https://code.claude.com/docs/en/common-workflows)

### Finding 15: No thinking strategy per complexity tier

- **Current**: Complexity Triage (L80-89) only determines template sections and
  research. No thinking depth guidance.
- **Official**: `ultrathink` keyword sets effort to high for Opus/Sonnet 4.6.
  `/effort` command controls session-wide thinking depth (low/medium/high/max).
- **Gap**: Deep tier benefits from higher thinking effort, but skill doesn't guide it.
- **Proposal**: Add "Thinking Strategy" column to Complexity Triage table. Deep tier
  can suggest `/effort high` or reasoning guidance prompts.

### Finding 16: Plan Mode ↔ Extended Thinking relationship unexplained

- **Current**: L74 just calls `EnterPlanMode`. No explanation of what Plan Mode
  does to thinking behavior.
- **Official**: Plan Mode is a read-only custom prompt mode. Extended thinking is
  enabled by default with adaptive depth. They are independent features.
- **Gap**: Users may think Plan Mode enables deeper thinking. It doesn't.
- **Proposal**: Add brief note in Plan Mode Transition section clarifying the
  relationship. For Deep tier, add reasoning guidance prompt after entering
  Plan Mode.

### Finding 17: Thinking-related anti-patterns missing

- **Current**: No guidance about thinking triggers in generated specs.
- **Official**: "think", "think hard" are regular prompt instructions on Claude 4.6,
  NOT thinking budget allocators. "Prefer general instructions over prescriptive
  steps."
- **Gap**: Users' raw prompts may include "think step by step" — the skill should
  know to remove or replace these with outcome-oriented guidance.
- **Proposal**: Add anti-pattern: "Do not include 'think step by step' or 'think
  hard' in refined specs — prefer outcome-oriented guidance."

### Finding 18: Interleaved thinking not leveraged in research protocol

- **Current**: Domain Research Protocol (L205-237) runs up to 3 WebSearch queries
  with no reflection guidance between searches.
- **Official**: Adaptive thinking enables interleaved thinking between tool calls.
  "After receiving tool results, carefully reflect on their quality and determine
  optimal next steps."
- **Gap**: Sequential blind searches miss the opportunity to refine queries based
  on prior results.
- **Proposal**: Add to Research Protocol: "After each WebSearch result, reflect on
  findings before deciding the next query."

## Axis ⑤: Subagent Handoff Patterns

Sources:
- [Subagents docs](https://code.claude.com/docs/en/sub-agents)
- [Skills docs](https://code.claude.com/docs/en/skills)
- [Multi-agent coordination patterns](https://claude.com/blog/multi-agent-coordination-patterns)

### Finding 19: Handoff mechanism ambiguous (Skill tool vs Agent tool)

- **Current**: L337-345 says "invoke subagent-dev skill" without specifying how.
- **Official**: Skills run inline via Skill tool (inheriting context). Subagents
  run in isolation via Agent tool. `context: fork` runs skills in isolated context.
- **Gap**: If subagent-dev is invoked via Skill tool, it inherits all REFINE phase
  context — contradicting subagent-dev's own principle of fresh context per task.
- **Proposal**: Make handoff explicit: "Write plan to file → tell user to run
  `/subagent-dev <plan-file-path>`. Or configure subagent-dev with `context: fork`."

### Finding 20: `isolation: worktree` not mentioned for parallel dispatch

- **Current**: subagent-dev warns "parallel implementation subagents = conflicts"
  as Red Flag. No mitigation.
- **Official**: `isolation: worktree` gives each subagent an isolated repo copy.
  Worktree auto-cleaned if no changes.
- **Gap**: Official feature directly solves the file conflict problem. Currently
  parallel implementation is forbidden unnecessarily.
- **Proposal**: Add `isolation: worktree` guidance for parallel implementation tasks.
  (Note: this is primarily a subagent-dev improvement, mentioned here for handoff
  context awareness.)

```markdown
### Claude 4.6 Notes

When the refined spec will be executed by Claude 4.6 (the default):

- **No prefilling**: Prefilled assistant responses are no longer supported.
  Use explicit instructions instead.
- **Overengineering tendency**: Opus 4.6 tends to create extra files, add
  unnecessary abstractions, and build in flexibility that wasn't requested.
  Include clear scope boundaries in `<task>`: "implement only what is specified,
  no additional features or abstractions."
- **Overthinking tendency**: Opus 4.6 does significantly more upfront exploration
  than previous models. If the raw prompt contains "be thorough" or "think about
  everything", soften or remove — it causes over-exploration.
- **General > prescriptive**: "Think thoroughly about the tradeoffs" produces
  better reasoning than a hand-written step-by-step analysis plan.
  In `<task>`, state goals and constraints; let the model determine the approach.
- **Scope control**: Refined specs should explicitly state what is OUT of scope,
  not just what is in scope. This counteracts the overengineering tendency.
```
