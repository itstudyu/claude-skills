# Workflow Blueprint Update

Incremental update of existing workflow documentation. Detects changes via
git diff, re-analyzes only affected workflows, and keeps docs in sync.

<HARD-GATE>
No speculation. No guessing.
Same rules as /workflow-blueprint: every participant, arrow, and step must
reference actual code you read. If you cannot trace a call to file:line,
write "not traced" — never infer from framework conventions or author intent.
</HARD-GATE>

## When to Run

- After code changes that touch routes, controllers, services, or repositories
- Before PR review to verify workflow documentation is current
- When `docs/workflow/INDEX.md` exists and may be stale
- When the user asks what changed in a specific workflow

## Step 0: Validate Existing Documentation

### 0-1. Check for INDEX.md

```
If docs/workflow/INDEX.md does NOT exist:
  → "No existing workflow docs found. Run /workflow-blueprint first
     to generate the initial analysis."
  → STOP. Do not proceed.
```

### 0-2. Read INDEX.md

Parse the INDEX.md to extract:
- List of all documented workflows with file paths
- Last full analysis date
- Tech stack and architecture pattern

### 0-3. Build Reverse File Map

Read each workflow document's **Participants** table to build a reverse map:

```
{
  "src/controllers/auth.controller.ts": ["post-api-auth-login", "post-api-auth-register"],
  "src/services/auth.service.ts": ["post-api-auth-login", "post-api-auth-register"],
  "src/services/user.service.ts": ["get-api-users-id", "put-api-users-id"],
  "src/repositories/user.repository.ts": ["post-api-auth-login", "get-api-users-id"],
  ...
}
```

This map is the key linkage: when a file changes, this tells us which
workflows are affected.

## Step 1: Detect Changes

### 1-1. Determine Change Scope

Ask the user or auto-detect:

| Scope | How to Detect | Command |
|-------|---------------|---------|
| Since last analysis | Read "Last analyzed" date from INDEX.md | `git diff --name-only --since="YYYY-MM-DD"` |
| Since specific commit | User provides commit hash | `git diff --name-only <commit>..HEAD` |
| Current uncommitted | User just made changes | `git diff --name-only` + `git diff --name-only --staged` |
| PR changes | User is reviewing a PR | `git diff --name-only <base-branch>...HEAD` |

If the user does not specify, default to "since last analysis date."

### 1-2. Categorize Changed Files

Run the git diff command and categorize results:

```
Changed Files: [total count]

Routes/Controllers: [count]
  - src/controllers/auth.controller.ts (modified)
  - src/routes/booking.routes.ts (new)

Services: [count]
  - src/services/auth.service.ts (modified)

Repositories/Models: [count]
  - src/repositories/user.repository.ts (modified)

Middleware/Guards: [count]
  - (none)

Other (config, tests, etc.): [count]
  - (not relevant to workflow docs)
```

### 1-3. Quick Exit Check

```
If 0 workflow-relevant files changed:
  → "No workflow-related files changed since last analysis.
     docs/workflow/ is up to date."
  → STOP.
```

## Step 2: Map Changes to Workflows

### 2-1. Match via Reverse File Map

For each changed file, look up the reverse map from Step 0-3:

```
Changed: src/services/auth.service.ts
  → Affects: post-api-auth-login, post-api-auth-register

Changed: src/repositories/user.repository.ts
  → Affects: post-api-auth-login, get-api-users-id, put-api-users-id
```

### 2-2. Detect New Entry Points

For files NOT in the reverse map, check if they are new entry points:
- New route/controller files → scan for endpoint definitions
- New event handlers, queue consumers, cron jobs

```
New entry point detected:
  - POST /api/v2/bookings → src/routes/booking.routes.ts:8
  → Will create new workflow document
```

### 2-3. Detect Deleted Entry Points

Check if any files referenced in existing workflow Participants tables
have been deleted:

```
Deleted: src/controllers/tours.controller.ts
  → Workflow "get-api-tours-search" references this file
  → Will mark as DEPRECATED
```

### 2-4. Present Change Summary

```
Workflow Update Plan:

UPDATE (re-analyze):
  - post-api-auth-login.md     (auth.service.ts changed)
  - post-api-auth-register.md  (auth.service.ts changed)
  - get-api-users-id.md        (user.repository.ts changed)

NEW (full analysis):
  - post-api-v2-bookings.md    (new endpoint detected)

DEPRECATE:
  - get-api-tours-search.md    (tours.controller.ts deleted)

Proceed?
```

Wait for user confirmation before modifying files.

## Step 3: Re-Analyze Affected Workflows

### 3-1. For UPDATED workflows

Re-run the full trace for each affected workflow using the same process
as `/workflow-blueprint` Steps 3-4:

1. Read the entry point (route/controller)
2. Trace through middleware → controller → service → repository → DB
3. Generate new Mermaid sequence diagram
4. Rebuild Participants table
5. Rebuild Data Touched table
6. Update External Calls and Error Paths

Compare with the existing document to identify what actually changed
(for the changelog entry).

### 3-2. For NEW workflows

Run full analysis from scratch — same as `/workflow-blueprint` Steps 3-5.
Use the standard workflow document template.

### 3-3. For DEPRECATED workflows

Do NOT delete the file. Instead, add a deprecation notice at the top:

```markdown
> **DEPRECATED** as of YYYY-MM-DD
> Reason: Entry point file [file path] was deleted.
> This workflow is no longer active in the codebase.
```

Update the workflow's status in INDEX.md to "deprecated."

## Step 4: Update Documents

### 4-1. Write Updated Workflow Files

For each updated workflow:
- Overwrite with the new analysis
- Update `> Last analyzed: YYYY-MM-DD` in the header
- Add a Changelog entry:

```markdown
## Changelog

| Date | Change | Details |
|------|--------|---------|
| YYYY-MM-DD | Updated | auth.service.ts: authenticate() now includes rate limiting check |
| [previous entries preserved] |
```

Preserve all existing Changelog entries. New entries go at the top.

### 4-2. Write New Workflow Files

Create new files using the standard template from `/workflow-blueprint`.
Include initial Changelog entry:

```markdown
## Changelog

| Date | Change | Details |
|------|--------|---------|
| YYYY-MM-DD | Created | Initial workflow analysis |
```

### 4-3. Update Deprecated Workflow Files

Add deprecation header and Changelog entry. Do not modify other content
(preserve for historical reference).

## Step 5: Update INDEX.md

Update `docs/workflow/INDEX.md`:

1. Update "Last full analysis" date
2. Add new workflows to the appropriate table (HTTP or Non-HTTP)
3. Mark deprecated workflows with ~~strikethrough~~ and "(deprecated)" label
4. Update Quick Stats counts

## Step 6: Generate Change Summary Report

Present to the user:

```
Workflow docs updated. Changes in docs/workflow/:

UPDATED (X workflows):
  - post-api-auth-login.md
    Changes: authenticate() now includes rate limiting (auth.service.ts:30)
    Diagram: 1 new participant added (RateLimiter)

  - get-api-users-id.md
    Changes: findById() query optimized with JOIN (user.repository.ts:15)
    Diagram: No structural change

NEW (X workflows):
  - post-api-v2-bookings.md
    Entry: POST /api/v2/bookings
    Participants: 5 | Tables: 2 | External calls: 1

DEPRECATED (X workflows):
  - get-api-tours-search.md
    Reason: tours.controller.ts deleted

Summary:
  - Workflows updated: X
  - Workflows added: X
  - Workflows deprecated: X
  - Total active workflows: X

All changelogs updated. INDEX.md refreshed.
```

## Edge Cases

| Case | Handling |
|------|---------|
| No git history available | Fall back to file modification times vs "Last analyzed" date |
| Shallow clone (no full history) | Use `git log --oneline -1` to check available history; ask user for commit range if needed |
| INDEX.md exists but is malformed | Re-read individual workflow files to rebuild the reverse map |
| Changed file not in any workflow | Check if it's a new entry point. If not, skip it (non-workflow file) |
| Massive change (50+ files) | Suggest running /workflow-blueprint for full re-analysis instead |
| Renamed/moved files | Git detects renames with `git diff --name-status`; map old → new path |
| Merge commits | Use `git diff <base>..HEAD` to see net changes, ignore intermediate states |
| Only test files changed | Skip — test files don't affect workflow documentation |

## Language and Output

- Respond in the user's language
- All generated/updated files in English
- Changelog entries in English
- Change summary report in user's language
