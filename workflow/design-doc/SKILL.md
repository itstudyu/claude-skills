# Design Doc Generator

Create a structured design document for a single feature. Each feature gets its own
file at `docs/specs/<feature-name>.md` following a fixed template. The document
captures everything someone needs to understand the feature: what it does, what data
it touches, what APIs it exposes, and how the UI works.

## When to Use

- Before modifying an undocumented feature
- When onboarding someone to a specific feature
- When the user wants to formalize how a feature works

## Step 1: Identify the Feature

Determine what feature to document. The user might say:
- "Write a design doc for the login feature"
- "Document the tour application system"
- A feature name from `docs/project-overview.md` (if it exists)

If the user is vague, check `docs/project-overview.md` for the feature list and ask
which one they want to document. If no overview exists, ask the user to name the
feature explicitly and point to representative files.

## Step 2: Analyze the Feature

Scan the codebase for everything related to this feature:

**Data Model:**
- Search ORM models, migration files, schema files for related tables
- Extract column names, types, constraints, relationships
- If no schema files: ask the user for table structure

**API Endpoints:**
- Search route/controller files for related endpoints
- Extract HTTP method, path, request/response shapes
- If no API files: ask the user or mark as N/A

**Business Logic:**
- Read service files, utility functions related to the feature
- Identify validation rules, state transitions, error handling
- Note any complex business rules or edge cases

**UI/Screen:**
- Search component/page files for related screens
- Identify which shared components are used
- Note routing paths and navigation flow
- If no frontend: mark as N/A

**Source Files:**
- Collect all file paths related to this feature

## Step 3: Interview for Gaps

After the code analysis, identify what's missing and ask the user. Common gaps:

- Table structure not in code (external DB, no ORM)
- Business rules that exist only in someone's head
- Validation rules not implemented yet
- External service integrations not visible in code

Ask concisely — bundle related questions into one AskUserQuestion call. Don't ask
about things you already found in the code.

## Step 4: Generate the Design Doc

Write to `docs/specs/<feature-name>.md` using this exact template. Every section
is present in every doc. Use "N/A" for sections that don't apply to this feature.

```markdown
# [Feature Name] Design Doc

> Auto-generated by /design-doc on YYYY-MM-DD
> Last updated: YYYY-MM-DD

## Overview
- **Description:** [What this feature does — 1-2 sentences]
- **Related screens:** [List of pages/routes, or N/A]
- **Status:** [Active / In Development / Planned]

## Data Model

| Table | Column | Type | Constraints | Notes |
|-------|--------|------|-------------|-------|
| users | id | SERIAL | PK | |
| users | email | VARCHAR(255) | UNIQUE, NOT NULL | Login identifier |
| users | role_id | INT | FK → roles | |

**Relationships:**
- users.role_id → roles.id (many-to-one)

(N/A if no database tables)

## API Endpoints

| Method | Path | Description | Auth | Request Body | Response |
|--------|------|-------------|------|-------------|----------|
| POST | /api/auth/login | User login | No | {email, password} | {token, user} |
| POST | /api/auth/logout | User logout | Yes | — | {success} |

(N/A if no API)

## Business Logic
- [Rule 1: description]
- [Rule 2: description]
- **Validation:** [field-level validation rules]
- **Error handling:** [how errors are handled and displayed]

## UI/Screen
- **Page:** [route path, e.g., /login]
- **Components used:** [list of shared/common components]
- **Figma:** [URL if available, otherwise N/A]
- **User flow:** [Step 1 → Step 2 → Step 3]

(N/A if no frontend)

## Files
| Path | Purpose |
|------|---------|
| src/auth/auth.service.ts | Authentication logic |
| src/auth/auth.controller.ts | API endpoints |
| src/pages/login/login.component.ts | Login page UI |
```

## Step 5: Commit and Report

After generating the document:

1. Create `docs/specs/` directory if it doesn't exist
2. Save the file
3. Offer to commit: "Design doc saved to `docs/specs/<feature>.md`. Want me to commit it?"

## Multiple Features

If the user wants to document several features at once, process them one at a time.
After each, ask: "Done with [feature]. Move on to the next one?"

This keeps each document focused and gives the user a chance to review and correct
before moving on. Don't batch them — quality matters more than speed here.

## Principles

- **Read code, don't guess.** If something isn't in the codebase, ask rather than assume.
- **Fixed template, always.** Every doc has the same sections. N/A is fine.
- **One feature, one file.** Don't combine multiple features into one doc.
- **Concise but complete.** The doc should be scannable in 2 minutes but contain all
  critical details. Skip boilerplate and obvious things.
